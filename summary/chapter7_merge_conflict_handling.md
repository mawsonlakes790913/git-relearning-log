# Chapter 7 Summary: Handling Merge Conflicts

## 概要
ブランチ間で同じ箇所に異なる変更が加えられた場合に発生する「コンフリクト」の発生条件と、その解消手順を確認した。  
今回は「ゲーム差列」と「勝率列」を別ブランチで追加し、それらを統合する過程で意図的にコンフリクトを発生させ、解決までの流れを実践した。 :contentReference[oaicite:0]{index=0}

---

## ■ コンフリクトが発生する状況

同一ファイルの同一箇所に対して、異なるブランチで別々の変更を行い、それらをマージしようとするとコンフリクトが発生する。

今回のケースでは：

- 編集者：ゲーム差列を追加  
- 管理者：勝率列を追加  

これらが同じテーブル構造に対する変更だったため、衝突が発生した。 :contentReference[oaicite:1]{index=1}

---

## ■ コンフリクト発生までの流れ

1. 編集者が新しいブランチで変更を行い、プルリクエストを作成
2. その間に管理者が別ブランチで変更を行い、先にmainへマージ
3. 編集者がmainの変更を取り込もうとしてマージ
4. コンフリクト発生

---

## ■ コンフリクトの確認

git status

→ `both modified` と表示され、競合しているファイルが特定できる

---

## ■ コンフリクトの構造

コンフリクトが発生すると、対象ファイルには以下のような区切りが表示される：

\<<<<<<< HEAD
（現在のブランチの変更）
=======
（マージ元の変更）
\>>>>>>> main

- HEAD側：自分のブランチの変更  
- main側：取り込もうとしている変更  

---

## ■ コンフリクトの解消方法

1. 該当箇所を確認し、どの変更を採用するか決定する  
2. 必要に応じて両方の変更を統合する  
3. 不要な区切り記号（<<<<<<< など）を削除する  

今回のケースでは、ゲーム差と勝率の両方を残す形で統合した。 :contentReference[oaicite:2]{index=2}

---

## ■ マージの完了

git add -A  
→ 修正内容をステージング

git commit  
→ コンフリクト解消後のコミット（メッセージは自動生成）

git push origin ブランチ名  
→ リモートへ反映

---

## ■ 最終確認

プルリクエストの差分を確認し、両方の変更が反映されていることを確認する。  
問題なければそのままマージして完了。

---

## ■ 本質理解

- コンフリクトは異常ではなく、並行開発において自然に発生する現象である  
- Gitは自動で解決できない変更のみを人間に委ねる  
- 解消の本質は「どの変更を残すか」ではなく「どう統合するか」である
