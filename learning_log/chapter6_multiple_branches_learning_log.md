# Chapter 6 学習ログ：複数ブランチの並行運用

## 概要
本章では、複数のブランチを同時に扱いながら開発を進める方法を実践した。  
一つの作業を途中で中断し、別の作業を先に完了させた後、元の作業を再開してマージする流れを確認した。

---

## ■ 実施内容

- 打撃成績の拡張（update-season-batting-stats）
- 試合結果の修正（update-postseason-result）

上記2つの作業を別ブランチで並行して進めた。

---

## ■ 作業A：打撃成績の拡張（途中で中断）

### 実施内容

- ブランチ作成

    git switch -c update-season-batting-stats

- HTML構造を変更
  - 守備位置列を追加
  - スタメン9人分の枠を作成
  - 一部選手はデータ未入力のままにした

---

### 学び

- ブランチは「未完成の状態」でも保持できる
- 完成していなくてもcommit・pushして問題ない

---

### 気づき

- すべて完成させてからcommitする必要はない
- 作業を小さく区切って保存することで安全に中断できる

---

## ■ 作業B：試合結果の修正（先に完了）

### 実施内容

- mainブランチに戻る

    git switch main

- 新しいブランチを作成

    git switch -c update-postseason-result

- HTMLを修正
  - 勝敗表記をスコア形式に変更
  - 勝利チームのスコアを強調

---

### 学び

- mainに戻らずにブランチを作成すると、別ブランチの変更を引き継いでしまう
- ブランチ作成時の起点は非常に重要

---

## ■ マージと同期

### 実施内容

- 作業Bを先にPull Requestし、管理者がマージ
- 本体（upstream）の最新状態を取得

    git switch main
    git pull upstream main

---

### 学び

- マージ後もローカルは自動更新されない
- 明示的に最新状態を取得する必要がある

---

## ■ 作業Aの再開

### 実施内容

- 作業Aのブランチに戻る

    git switch update-season-batting-stats

- mainの変更を取り込む

    git merge main

- 未入力だった打撃成績をすべて完成させる

---

### 学び

- 別ブランチの変更は自動では反映されない
- mainを経由して取り込む必要がある

---

### 気づき

- 並行作業では「どのブランチがどの状態を基準にしているか」を常に意識する必要がある

---

## ■ 最終マージ

### 実施内容

- 作業Aをコミット・プッシュ
- Pull Requestを作成しマージ

---

## ■ 発生し得る問題と理解

### 問題1：ブランチの起点ミス

- mainに戻らずにブランチを作成すると、意図しない変更を含むブランチになる

---

### 問題2：最新状態の不一致

- upstreamの変更を取り込まないまま作業を進めると差分がずれる

---

## ■ 本質理解

- ブランチは「作業単位」で分けるもの
- 作業は途中で中断・再開できる
- 複数ブランチの統合はmainを基準に行う

---

## ■ 今回の到達点

- 複数ブランチを並行して運用できた
- 作業の優先順位に応じてマージ順を変更できた
- mainを中心としたブランチ統合の流れを理解できた
